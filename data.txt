## 1. Reduce Concurrency Safely

### Finding: `SimpleAsyncTaskExecutor` creates UNLIMITED threads

At RedeterminationNightlyConfiguration.java:

```java
.taskExecutor(this.fwTaskExecutor)
.taskExecutor(new SimpleAsyncTaskExecutor()).build();
```

The second `.taskExecutor()` call replaces the first. `SimpleAsyncTaskExecutor` spawns **one thread per partition with no upper bound**. If the DB has `PRTN_CNT=10`, all 10 partitions run simultaneously. The `thread.core.pool.size` and `thread.max.pool.size` properties have **zero effect** because `fwTaskExecutor` is never used.

### CHANGE 1A — Remove `SimpleAsyncTaskExecutor`, use managed pool

**Classification: Must do now**

**Class:** `RedeterminationNightlyConfiguration`
**Method:** `redeterminationNightlyStep()`
**File:** RedeterminationNightlyConfiguration.java

**Exact code edit:**

## 2. Add Retry for DB2 -913

### Finding: Current retry is completely non-functional

### Idempotency assessment of current code paths

Before recommending retry, I must check whether the writer operations are safe to re-execute:

| Operation | Code location | Idempotent? | Reasoning |
|-----------|--------------|-------------|-----------|
| `updateNextRedDateWithNoRealignFlag` | Writer line 79 | **Yes** | `UPDATE ... SET col=? WHERE ICI=? AND ... AND REDET_STRT_DT='9999-12-31'` — after first execution, WHERE clause no longer matches (REDET_STRT_DT is changed), so re-execution updates 0 rows harmlessly |
| `updateNextRedDateWithRealignFlag` | Writer line 75 | **Yes** | Same reasoning — WHERE includes `REDET_STRT_DT='9999-12-31'` |
| `caseLogSorDao.insert` | Writer lines 77, 92 | **No** | Could double-insert a case log row |
| `insertTwnEditMsgRecord` | Writer line 87 | **Half** | Guarded by `getTwnEditMessageData` check, but on retry the first insert may have committed |
| `processRecords` (split) | Writer line 90 | **Partially** | `InsertNewOpenEndedSorMultipleRecord.execute()` does `find()` before `insert()` — safe. `TerminateSorRecord.execute()` does `UPDATE SET PRD_END_DT=?` — idempotent if same value |

**Critical problem:** The writer does `connection.commit()` per item inside the loop. If item 3 of 5 in a chunk deadlocks, items 1-2 are already committed. Spring Batch retry replays the entire chunk, so items 1-2 would be re-processed. The `REDET_STRT_DT='9999-12-31'` WHERE clause protects the main update (0 rows on re-run), but `caseLogSorDao.insert` would double-insert.

**Classification: Not minimal / postpone**

**Why:** Safe retry requires either (a) removing the manual `connection.commit()` and ensuring chunk size = 1 from DB, or (b) wrapping each item in its own retry loop inside the writer. Option (a) changes transaction semantics and needs end-to-end testing. Option (b) requires adding Spring Retry as a dependency plus a new `RetryTemplate` bean. Both exceed "smallest safe code change."

**What would make it minimal:** If and only if you can confirm that `CHK_SZ = 1` in your batch framework DB table for this job, then removing the manual commit AND adding `DeadlockLoserDataAccessException.class` to the retry list would work. But I cannot verify that DB value from code alone.
