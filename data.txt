A) Death Master File (DMF) Processing & Automation Requirements
1.	DMF File Processing and Data Match: 
o	Create a Batch Job to extract data from the Quarterly DMF file and load it into the database table 
o	Implement a batch to perform data match against current Medicaid, SNAP, and TANF enrollees to identify potentially deceased individuals. 
2.	Death Match Crystal Report: 
o	Update the existing Death Match Crystal Report by including the data received from DMF file.
3.	Customer Communication – PDF Notice:
o	Implement a new Death Match Notice and send that notice to the Head of the Household and Authorized Representatives as applicable through state print and mailing facilities.
o	A separate Death Match notice template will be created in SmartComm
4.	Task Management:
o	Create tracking tasks in AMPS 
	Create new Task Type ‘Death Match’ in AMPS
5.	Case Management:
o	Automate the logic for taking an action based on program-specific criteria (SNAP, TANF, Medical) once the due date has passed, based on rules identified during design session.
o	Auto close the tasks in AMPS for cases that are processed by case workers and manual CLOG was entered after the Death Match notice issued
o	Auto-close the respective ‘Death Match’ tasks in AMPS for cases that are auto-processed.
o	If there is an error in the automated case closure then, create a new task for case worker to act upon the case
6.	Audit Trail: 
o	Insert a CLOG about the DMF Death Match when found.
o	Insert CLOGs in AMPS when tasks are created and notices are sent
o	Insert CLOGs indicating the action is taken for the case closure
7.	MMIS Integration:
o	Share information with MMIS about the closure due to Death Match in existing MMIS file 
8.	Reports: 
o	Provide up to 3 ad-hoc reports to support tracking and reporting, based on the requirements identified during design sessions
B) CMS Concurrent Enrollment File Automation Requirements
1.	Concurrent Enrollment File Processing and Filtering Rules: 
o	Implement a batch to process the CMS Concurrent Enrollment File and apply the following filtering rules: 
	Compare the CMS Concurrent Enrollment file to the most recent PARIS file to exclude individuals who were already identified and acted upon through PARIS, with an exception to not exclude “Homeless” designation 
	Exclude individuals who don’t have active Medicaid coverage
	Exclude Individuals if residency is verified after the Concurrent Enrollment file receipt date
	Exclude Individuals failed Ex-parte within 30 days from the Concurrent Enrollment file receipt date
2.	Customer Communication – PDF Notice:
o	Implement a new Concurrent Enrolment Notice and send that to Head of the Household and Authorized Representatives as applicable through state print and mailing facilities.
o	A new Concurrent Enrolment Notice template will be created in SmartComm
3.	Task management: 
o	Create tracking tasks in AMPS 
	Create new Task Type ‘Concurrent Enrollment’ in AMPS

4.	Case Management:
o	Automate the logic for taking an action to terminate the FMC coverage for individuals who do not respond to the PDF notice. 
o	Auto-close the respective ‘Concurrent Enrollment’ tasks in AMPS for cases that are auto-processed
o	Auto close the tasks in AMPS for cases that are processed by case workers and manual CLOG was entered after the Concurrent Enrollment notice issued. 
o	If there is an error in the automated case closure then, create a new task for case worker to act upon the case
5.	Audit Trail: 
o	Insert CLOGs in AMPS when tasks are created and notices are sent
o	Insert CLOGs indicating the action is taken for the case closure
6.	MMIS Integration:
o	Share information with MMIS about the closure due to Concurrent Enrollment in existing MMIS file
7.	Reports: Provide up to 3 ad-hoc reports for tracking 
o	Individual-level counts and final outcomes with source attribution (Concurrent Enrollment vs PARIS vs renewal determination) for DSS/NVHA Reporting and CMS reconciliation
