## 1. Root Cause Diagnosis

### Why ICI overlap happens (RC1)

In the partitioner at RedeterminationNightlyPartitioner.java:

```java
final List<List<Long>> iciSubLists = Lists.partition(iciList, partitionSize);
for (final List<Long> icis : iciSubLists) {
    value.putLong(MIN_VALUE, Collections.min(icis));
    value.putLong(MAX_VALUE, Collections.max(icis));
}
```

This stores only the **min and max ICI** of each sublist. The reader then uses `WHERE T1.ICI >= ? AND T1.ICI <= ?`. If the ICI list is `[100, 200, 300, 400]` and partition size is 2, you get partition 0 = `[100,200]` (min=100, max=200) and partition 1 = `[300,400]` (min=300, max=400). This seems safe only if ICIs are contiguous. But the real problem is deeper:

**A single ICI can appear as multiple rows** (one per program case type: AF, TN, MA, TC, TL). The reader executes 3 separate queries for TN, TC, and "all others", each returning the same ICI with different program types. Two partitions whose ICI ranges overlap by even one ICI will both process that ICI's individuals (same UPIs) in parallel.

### Why static state causes corruption (RC2)

`RederminationSplitModel` has **all static fields** — `prdBeginDate`, `prdEndDate`, `splitEndDate`, `splitBgnDate`. In RedeterminationNightlyService.java:

## 2. Most Probable Deadlock Scenario in Current Code

Here is the exact scenario happening in production:

```
Thread-1 (Partition 0, ICI=5000, program=AF):
  1. UPDATE TWNREDET_RCRT_DTLS WHERE ICI=5000 AND PRGM_CASE_TYPE='AF'  → acquires X-lock on row R1
  2. Loop over individuals (UPI=8001, 8002, 8003, 8004)
     For UPI=8001:
       updateJobIncome → UPDATE TWNJOB_INCM_DTLS WHERE UPI=8001  → acquires X-lock on rows J1,J2
       updateAddress   → UPDATE TWNPERSON_ADRS_HST WHERE UPI=8001  → WAITING for lock on A1

Thread-2 (Partition 1, ICI=5001, program=TN):
  1. UPDATE TWNREDET_RCRT_DTLS WHERE ICI=5001 AND PRGM_CASE_TYPE='TN'  → acquires X-lock on row R2
  2. UPI=8001 happens to be a member of BOTH ICI=5000 and ICI=5001
     For UPI=8001:
       updateAddress   → UPDATE TWNPERSON_ADRS_HST WHERE UPI=8001  → acquires X-lock on A1
       updateJobIncome → UPDATE TWNJOB_INCM_DTLS WHERE UPI=8001    → WAITING for lock on J1

  *** DEADLOCK: Thread-1 holds J1, waits for A1. Thread-2 holds A1, waits for J1. ***
```

This is **lock inversion** caused by:
1. Same UPI (8001) being a member of two different ICIs processed in parallel
2. Different table update ordering: AF updates `JobIncome → Address`, while TN updates `Address → JobIncome` (see RedeterminationSplitDataService.java)

**Answer to your explicit question: YES, the deadlocks are overwhelmingly caused by parallel partitions processing overlapping ICI/UPI rows rather than by DB2 alone.** DB2 is correctly detecting a genuine application-level lock cycle. The database is not at fault.

### Fix 3B: Add retry for SQLCODE -913

In RedeterminationNightlyConfiguration.java, the current retry is only for `BatchRetryRecordException`. Add DB2 deadlock retry:

```java
// Create a new exception class
package gov.nv.dwss.redetermination.retry;

import org.springframework.classify.Classifier;
import org.springframework.retry.RetryPolicy;
import org.springframework.retry.policy.NeverRetryPolicy;
import org.springframework.retry.policy.SimpleRetryPolicy;
import org.springframework.dao.DeadlockLoserDataAccessException;
import org.springframework.dao.CannotAcquireLockException;

public class Db2DeadlockRetryPolicy implements Classifier<Throwable, RetryPolicy> {
    
    private final int maxRetries;
    
    public Db2DeadlockRetryPolicy(int maxRetries) {
        this.maxRetries = maxRetries;
    }
    
    @Override
    public RetryPolicy classify(Throwable throwable) {
        if (isDeadlockOrTimeout(throwable)) {
            return new SimpleRetryPolicy(maxRetries);
        }
        return new NeverRetryPolicy();
    }
    
    private boolean isDeadlockOrTimeout(Throwable t) {
        while (t != null) {
            if (t instanceof DeadlockLoserDataAccessException 
                || t instanceof CannotAcquireLockException) {
                return true;
            }
            String msg = t.getMessage();
            if (msg != null && (msg.contains("SQLCODE=-913") || msg.contains("SQLSTATE=57033"))) {
                return true;
            }
            t = t.getCause();
        }
        return false;
    }
}
```

## 5. Recommended Partitioning Model

### Current problem

The partitioner uses `minValue`/`maxValue` ranges. The reader re-queries all ICIs in that range. An individual (UPI) can be a member of multiple ICIs. Two partitions processing different ICIs can touch the same UPI's records.

### Recommended: Partition by ICI with exact ownership

**Replace range-based partitioning with exact ICI list assignment:**

```java
@Override
public Map<String, ExecutionContext> partition(final int gridSize) {
    Map<String, ExecutionContext> result = new HashMap<>();
    
    try {
        // ... existing date initialization code ...
        
        List<Long> iciList = /* existing ICI retrieval logic */;
        
        // Sort ICIs deterministically
        Collections.sort(iciList);
        
        LOG.info("Total ICIs to process: {}", iciList.size());
        
        if (iciList.isEmpty()) {
            ExecutionContext value = new ExecutionContext();
            result.put("partition0", value);
            value.putString("iciList", "");
            return result;
        }
        
        // Divide into exact sublists - no range overlap possible
        List<List<Long>> partitions = Lists.partition(iciList, 
            Math.max(1, iciList.size() / gridSize));
        
        for (int i = 0; i < partitions.size(); i++) {
            ExecutionContext ctx = new ExecutionContext();
            // Store the EXACT list of ICIs, not a range
            String iciCsv = partitions.get(i).stream()
                .map(String::valueOf)
                .collect(Collectors.joining(","));
            ctx.putString("iciList", iciCsv);
            ctx.putLong("partition", i);
            result.put("partition" + i, ctx);
        }
        
    } catch (Exception e) {
        LOG.error("Partitioner failed", e);
        throw new RuntimeException(e);
    }
    return result;
}
```

### Why this eliminates overlap

Each partition gets a **disjoint set of ICIs**. No two partitions ever process the same ICI. Since the `TWNREDET_RCRT_DTLS` update is keyed by `(ICI, PRGM_CASE_TYPE)`, there's zero contention on that table.

### Remaining risk: UPI shared across ICIs

Even with disjoint ICI partitioning, a UPI (individual) can theoretically belong to two different ICIs (if a person is on two different cases). For the `TWNPERSON_ADRS_HST` table keyed by `(UPI, PRD_BEG_TS)`, this is still a contention risk. Mitigation options:

**Option A (recommended):** Also group ICIs by their member UPIs. Before partitioning, build a UPI→ICI graph: any ICIs sharing a UPI must go to the same partition.

```java
/**
 * Groups ICIs that share UPIs into the same partition to prevent
 * concurrent updates on shared individuals.
 */
private List<Set<Long>> groupIcisBySharedUpi(List<Long> iciList) {
    // Query: SELECT ICI, UPI FROM TWNCASE_MEMBER_HST WHERE ICI IN (...)
    Map<Long, Set<Long>> iciToUpis = loadIciToUpiMapping(iciList);
    
    // Union-Find to group ICIs sharing any UPI
    UnionFind<Long> uf = new UnionFind<>();
    Map<Long, Long> upiToFirstIci = new HashMap<>();
    
    for (Map.Entry<Long, Set<Long>> entry : iciToUpis.entrySet()) {
        long ici = entry.getKey();
        for (long upi : entry.getValue()) {
            if (upiToFirstIci.containsKey(upi)) {
                uf.union(ici, upiToFirstIci.get(upi));
            } else {
                upiToFirstIci.put(upi, ici);
            }
        }
    }
    
    // Collect groups
    Map<Long, Set<Long>> groups = new HashMap<>();
    for (long ici : iciList) {
        long root = uf.find(ici);
        groups.computeIfAbsent(root, k -> new LinkedHashSet<>()).add(ici);
    }
    return new ArrayList<>(groups.values());
}
```

**Option B (simpler):** Keep disjoint ICI partitioning + add retry for the rare cross-ICI UPI collision.

I recommend **Option B** for now (simpler), moving to Option A only if you still see deadlocks after the other fixes.

### 6B: The `REDET_STRT_DT = '9999-12-31'` predicate problem

This is a **hot-spot pattern**. All unprocessed records share the same sentinel value `'9999-12-31'`. If the index leading columns are `(ICI, PRGM_CASE_TYPE)`, then the sentinel isn't a problem because the ICI narrows it to 1-2 rows. But if the index has `REDET_STRT_DT` as a leading column, **all unprocessed records cluster together**, creating page-level lock contention.

**Recommendation:** Ensure `REDET_STRT_DT` is NOT a leading index column. It should be the last column in the composite index:

```sql
-- GOOD: ICI first, sentinel last
CREATE INDEX IX_REDET_LOOKUP 
    ON TWNREDET_RCRT_DTLS (ICI, PRGM_CASE_TYPE, NEXT_REDET_DT, REDET_STRT_DT);

-- BAD: sentinel first causes hot-spot
-- CREATE INDEX IX_REDET_BAD ON TWNREDET_RCRT_DTLS (REDET_STRT_DT, ICI, ...);
```

### 6C: DB2 lock escalation settings

Check whether DB2 is escalating row locks to table locks:

```sql
-- Check current lock escalation threshold
SELECT * FROM SYSIBMADM.DBCFG WHERE NAME = 'LOCKLIST' OR NAME = 'MAXLOCKS';
```

If `MAXLOCKS` is too low (e.g., 10%), DB2 escalates to table locks when a single connection holds too many row locks. With 4 individuals × 15+ tables per case, a single chunk could easily hit this. **Recommend:**

```sql
-- Increase to prevent premature escalation
UPDATE DB CFG USING LOCKLIST 50000 MAXLOCKS 40;
```

### 6D: DB2 lock timeout

```sql
-- Increase lock timeout to give retries more room
UPDATE DB CFG USING LOCKTIMEOUT 30;  -- 30 seconds instead of default
```

## 7. Retry Strategy with Java/Spring Code

### 7A: Spring Retry with exponential backoff + jitter

Add to your pom.xml:

```xml
<dependency>
    <groupId>org.springframework.retry</groupId>
    <artifactId>spring-retry</artifactId>
</dependency>
```

Create a retry template for DB2 deadlocks:

```java
package gov.nv.dwss.redetermination.retry;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.dao.CannotAcquireLockException;
import org.springframework.dao.DeadlockLoserDataAccessException;
import org.springframework.retry.backoff.ExponentialRandomBackOffPolicy;
import org.springframework.retry.policy.SimpleRetryPolicy;
import org.springframework.retry.support.RetryTemplate;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class RetryConfig {

    @Bean("db2RetryTemplate")
    public RetryTemplate db2RetryTemplate() {
        RetryTemplate template = new RetryTemplate();

        // Retry up to 3 times for deadlock/timeout
        Map<Class<? extends Throwable>, Boolean> retryableExceptions = new HashMap<>();
        retryableExceptions.put(DeadlockLoserDataAccessException.class, true);
        retryableExceptions.put(CannotAcquireLockException.class, true);
        SimpleRetryPolicy retryPolicy = new SimpleRetryPolicy(3, retryableExceptions);
        template.setRetryPolicy(retryPolicy);

        // Exponential backoff with jitter: 500ms → 1s → 2s (±random)
        ExponentialRandomBackOffPolicy backOff = new ExponentialRandomBackOffPolicy();
        backOff.setInitialInterval(500L);
        backOff.setMultiplier(2.0);
        backOff.setMaxInterval(5000L);
        template.setBackOffPolicy(backOff);

        return template;
    }
}
```

### 7B: Use the retry template in the writer

```java
package gov.nv.dwss.redetermination.writer;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.retry.support.RetryTemplate;
// ... other imports ...

public class RedeterminationNightlyWriter implements ItemWriter<RedeterminationNightlyDto> {

    @Autowired
    @Qualifier("db2RetryTemplate")
    private RetryTemplate retryTemplate;

    @Override
    public void write(final List<? extends RedeterminationNightlyDto> items) throws Exception {
        for (final RedeterminationNightlyDto rd : items) {
            final int ici = rd.getRedeterminationNightly().getIci();
            final String program = rd.getRedeterminationNightly().getProgramCaseType();

            retryTemplate.execute(retryContext -> {
                if (retryContext.getRetryCount() > 0) {
                    LOG.warn("Retry #{} for ICI={}, program={}", 
                        retryContext.getRetryCount(), ici, program);
                }
                processOneItem(rd);
                return null;
            });
        }
    }

    private void processOneItem(RedeterminationNightlyDto rd) throws Exception {
        RedeterminationNightly redet = rd.getRedeterminationNightly();

        // 1. Update redetermination details (table 1)
        if (redet.getReassignFlag() == 'Y') {
            redeterminationNightlyService.updateNextRedDateWithRealignFlag(redet);
            caseLogSorDao.insert(rd.getCaseLogForRealignFlag());
        } else {
            redeterminationNightlyService.updateNextRedDateWithNoRealignFlag(redet);
        }

        // 2. TWN Edit message
        Integer upi = redeterminationSplitDataService.getUPI(redet);
        List<SorRecord> editMsgList = redeterminationSplitDataService
            .getTwnEditMessageData(redet, upi);
        if (CollectionUtils.isEmpty(editMsgList)) {
            redeterminationSplitDataService.insertTwnEditMsgRecord(redet, upi);
        }

        // 3. Split records (table updates in deterministic order)
        if (!redet.getProgramCaseType().equalsIgnoreCase("FS")) {
            redet = updateIndicator(redet);
            redeterminationSplitDataService.processRecords(redet);
        }

        // 4. Case log
        if (!ObjectUtils.isEmpty(rd.getCaseLogForRDBatch())) {
            caseLogSorDao.insert(rd.getCaseLogForRDBatch());
        }
    }
}
```

| `updateNextRedDateWithRealignFlag` | Yes | Yes | Same reasoning |
| `sorDao.update()` (terminate/split) | Yes | Yes | SET PRD_END_DT = specific value WHERE key matches; idempotent |

## 9. What Metrics/Logging to Add to Prove the Fix

### 9A: Add deadlock detection logging

```java
package gov.nv.dwss.redetermination.listener;

import org.springframework.batch.core.listener.ChunkListenerSupport;
import org.springframework.batch.core.scope.context.ChunkContext;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Component;

import java.util.concurrent.atomic.AtomicInteger;
import java.util.concurrent.atomic.AtomicLong;

@Component
public class DeadlockMetricsListener extends ChunkListenerSupport {
    
    private static final Logger LOG = LoggerFactory.getLogger(DeadlockMetricsListener.class);
    
    private static final AtomicInteger deadlockCount = new AtomicInteger(0);
    private static final AtomicInteger retryCount = new AtomicInteger(0);
    private static final AtomicLong totalWriteTimeMs = new AtomicLong(0);
    private static final AtomicInteger chunkCount = new AtomicInteger(0);
    
    public static void recordDeadlock() { deadlockCount.incrementAndGet(); }
    public static void recordRetry() { retryCount.incrementAndGet(); }
    public static void recordWriteTime(long ms) { totalWriteTimeMs.addAndGet(ms); }
    
    @Override
    public void afterChunk(ChunkContext context) {
        int chunk = chunkCount.incrementAndGet();
        if (chunk % 100 == 0) {
            LOG.info("METRICS: chunks={}, deadlocks={}, retries={}, avgWriteMs={}",
                chunk, deadlockCount.get(), retryCount.get(),
                chunk > 0 ? totalWriteTimeMs.get() / chunk : 0);
        }
    }
    
    @Override
    public void afterChunkError(ChunkContext context) {
        LOG.warn("METRICS: chunk error. Total deadlocks so far: {}", deadlockCount.get());
    }
}
```

**After Phase 7**, you should be able to run with `grid.size=8` (8 parallel partitions) and see zero deadlocks. The total batch time for 100k cases should drop from hours to ~30-45 minutes.

### Summary

Your deadlocks are **100% an application design problem**, not a DB2 problem. The three root causes in priority order are:

1. **Static mutable state** (`RederminationSplitModel`, `GenericSplitRecord`) → threads corrupt each other's dates and record lists
2. **Range-based partitioning** → overlapping ICI ranges means the same UPI gets updated by 2+ threads simultaneously
3. **Non-deterministic table update ordering** → different program types update tables in different sequences, causing lock inversion

Fix all three and deadlocks disappear entirely. The retry mechanism is a safety net for the rare edge case of cross-ICI shared UPIs.
